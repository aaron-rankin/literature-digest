# Subtask 12 — Report UI overhaul

**To-do:** new — make generated HTML reports sleeker, more digestible, and more presentable.
**Decisions:** D7 (group by search term), D9 (borderline flag) — this subtask builds on those.
**Depends on:** 05 (matched_terms + report grouping), 08 (borderline flag) ideally; can run design explorations in parallel.

## Goal

Upgrade the current plain HTML reports (`templates/area.html.j2` and `templates/index.html.j2`) into a polished, readable digest that a coach or performance director can scan quickly. The reports should feel intentional: clear hierarchy, consistent typography, helpful micro-interactions, and a layout that works on desktop, tablet, and mobile.

## Current pain points

- Walls of text — abstracts and action points are hard to scan.
- No visual distinction between score bands or categories.
- No grouping by search term, so cross-term overlap is invisible.
- No "at a glance" summary of what changed since the last run.
- Basic styling that looks like a prototype, not a finished product.

## Design directions to decide (grill session needed)

1. **Information density**
   - Collapsible abstracts? Expand-on-click action points? Or show everything by default?
   - One card per article, or a denser list view?

2. **Visual hierarchy**
   - Score badge sizes/colors by band (e.g. 80-100 strong, 60-79 moderate, borderline flagged).
   - Category chips with icons or color coding (`directly actionable`, `monitoring`, `background`).
   - Search-term pills / filter chips.

3. **Grouping & navigation**
   - Group area report under per-term headings (subtask 05).
   - Table of contents / sticky sidebar for jumping between terms.
   - Articles matched by multiple terms appear under each term with a "also matched by X" indicator.

4. **Digestibility helpers**
   - One-line "key takeaway" at the top of each card.
   - Summary sentence per article generated by the LLM, or just styled rationale?
   - "Why this matters" section using the rationale.

5. **Index page**
   - Area cards with counts, trend arrows (more/less retained vs last run), last-run date.
   - Quick filters: "actionable only", "new this run".

6. **Responsive & accessibility**
   - Mobile-first card layout.
   - Print stylesheet for PDF export (even though PDF generation is out of scope, print-to-PDF should look decent).
   - Color-blind safe category indicators (not color-only).

7. **Optional "nice to haves"**
   - Dark mode toggle.
   - Sort by score, date, or category.
   - Copy citation / DOI button.

## Proposed implementation

1. **Design mock-up first**
   - Produce 2-3 quick options (wireframes or HTML prototypes) before committing.
   - Review against the decision questions above.

2. **Template refactor**
   - Rewrite `templates/area.html.j2` with:
     - Header: area name, run metadata, summary counts.
     - Per-term sections.
     - Article cards with score, category, title, citation, DOI link, abstract, rationale, action points.
     - Borderline tag (subtask 08).
   - Rewrite `templates/index.html.j2` as a dashboard with area cards.

3. **CSS-only styling**
   - Keep everything self-contained in the template `<style>` block so reports are portable (no external build step).
   - Use CSS variables for theming.

4. **Report context updates**
   - `report.py`: pass any extra context needed (e.g. per-term article lists, borderline band, previous run counts for trends).
   - `AreaIndexRow`: add fields for trend/comparison if needed.

5. **Golden / snapshot tests**
   - Snapshot key HTML fragments so future styling refactors don't accidentally break structure.

- [ ] Articles are grouped by search term with cross-term overlap visible.
- [ ] Borderline articles are visually flagged.
- [ ] Reports are readable on a phone without horizontal scrolling.
